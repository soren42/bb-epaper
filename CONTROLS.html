bb-epaper · documentation

Controls & Interface Reference

How to set up, configure, and live with the device.

At a Glance

bb-epaper is a two-piece system. A Raspberry Pi service renders an 800×480 monochrome frame and serves it over HTTP; a Seeed XIAO ESP32-C3 on the wall fetches that frame every few minutes and blits it to a 7.5" e-paper panel. Everything you configure lives on the Pi — the device is a thin client.

What you touch

  • BOOT button on the XIAO — runtime gesture to re-enter setup
  • Captive portal at http://192.168.4.1 — first-time WiFi config
  • Web UI at http://<pi-ip>:8081/ — day-to-day configuration

What you don't touch

  • The firmware — no reflash needed to change layout, pages, cadence, or data
  • The panel — no buttons, no controls; output only
  • The Driver Board's user buttons — present but not wired into firmware

The Device

The XIAO snaps into a Seeed XIAO ePaper Driver Board V2 (P/N 6374), which routes SPI plus the four control lines (BUSY, RST, DC, CS) to the 24-pin FPC connector on the 7.5" panel. The Driver Board has its own buttons; the bb-epaper firmware does not use them. The only physical control that matters for bb-epaper is the XIAO's BOOT button.

Physical Control · The BOOT Button

BOOT is dual-purpose. Its meaning depends on when you press it.

During Reset USB Download Mode

Hold BOOT while pressing RESET (or while plugging in USB). The ESP32-C3 enters its ROM USB bootloader. The device will not run firmware; you can flash it from arduino-cli or esptool. Release once enumeration is done.

During Runtime Re-enter Setup

With the device already booted and running normally, hold BOOT for ≥ 2 seconds. The firmware notices, wipes saved WiFi credentials, paints a setup splash, and reboots into the bb-epaper-setup AP.

Try the gesture

Press and hold the simulated BOOT button below. The real device uses the same timing.

held: 0.00 s   threshold: 2.00 s
Idle. Press and hold to start.

Releasing before 2 s does nothing — the firmware resets the press counter. Releasing after 2 s commits: WiFi creds are erased, the panel paints "setup mode — rebooting into portal…", and the device restarts into the captive-portal AP.

First-Time Setup · The Captive Portal

On first boot the device has no WiFi credentials, so it opens an open access point named bb-epaper-setup. The same path is also taken after a BOOT-hold gesture.

Step by step

  1. From your phone or laptop, join the WiFi network bb-epaper-setup (open, no password).
  2. The captive portal should pop up. If it doesn't, browse to http://192.168.4.1.
  3. Tap Configure WiFi, pick your SSID, enter the password.
  4. Set BB-Terminal server URL to http://<pi-ip>:8081 (default http://10.5.2.20:8081).
  5. Tap Save. The device reboots, joins WiFi, pulls a frame, paints it.

The portal has a 3-minute timeout. If you don't save in time, the device reboots and tries to reconnect with whatever creds it had (or, on first boot, comes right back into the portal).

Day-to-Day · The Web UI

Once the device is online, everything else is configured from the Pi's web UI at http://<pi-ip>:8081/. Open it in any browser on the same network.

What each control does

  • refresh_seconds — how long the device sleeps between frame fetches. Default 300. Clamped 60–3600 by the firmware. Server-controlled: the new value reaches the device on its next heartbeat reply.
  • pages — comma-separated list of page names to rotate through. Built-ins: watch, cc. The device picks one each cycle: page = int(time.time() // refresh_seconds) % len(pages).
  • data_ttl_seconds — how long the Pi caches upstream market data before re-fetching. Independent from refresh_seconds; the rendered frame can be re-baked from cached data.

Changes apply on the next heartbeat round-trip — i.e., within one refresh_seconds window at worst.

What the Device Shows

Layout changes don't require a reflash — edit /usr/local/bb-epaper/renderer.py on the Pi and run sudo systemctl restart bb-epaper. The new layout will appear within one refresh_seconds window.

For Integrators · HTTP Endpoints

The Pi service exposes a small HTTP API at port 8081. Useful if you want to drive the device from outside the web UI, or write your own client.

MethodPathPurpose
GET / HTML web UI (the page mocked above).
GET /epaper/frame.bin 48 000 bytes of 1-bpp packed pixels (800 × 480, MSB-first, row-major). Response header X-Epaper-Page names the page baked into this frame.
GET /epaper/frame.png?page=<name> The same frame rendered as PNG, for web preview / debugging.
GET /epaper/config Current config as JSON: {refresh_seconds, pages, data_ttl_seconds, host, port}.
POST /epaper/config Update config. Body is a partial JSON object; missing fields are kept.
POST /epaper/heartbeat Device check-in. Body {fw, rssi, page}. Reply {refresh_seconds} — the device honors this on its next sleep.
GET /epaper/status Last heartbeat, current page, last error, cache age — for monitoring.

No auth. The service binds 0.0.0.0:8081, so it's reachable from anywhere on the LAN. Don't expose it to the internet without a reverse proxy.

Quick Troubleshooting

Blank panel after boot

Open the serial monitor at 115 200 baud (USB-C to the XIAO). WiFi or HTTP errors print there. Confirm the Pi service is up: 

sudo systemctl status bb-epaper
journalctl -u bb-epaper -n 50

No bb-epaper-setup AP after BOOT-hold

Did you hold BOOT during reset instead of during runtime? Holding during reset puts the chip into USB download mode, not setup mode. Power-cycle and wait ~30 s — the firmware tries connecting before opening the AP.

Short read on frame.bin

The Pi service may be doing a cold fetch from upstream APIs (can take ~8 s). The firmware's HTTP timeout is 15 s, so cold cache is usually fine. On a flaky network, bump the timeout in fetchFrame().

Web UI unreachable

SSH to the Pi and confirm the service is serving: 

curl -I http://127.0.0.1:8081/epaper/frame.bin
sudo systemctl restart bb-epaper

Ghosting on the panel

The firmware forces a full refresh every cycle, so ghosting shouldn't accumulate. If it does, unplug the panel for a few seconds and re-seat the FPC connector. Some panels need an all-white frame to fully clear after long static periods.

Want to change WiFi networks

Hold BOOT for ≥ 2 s while the device is running. It wipes saved creds and reopens the captive portal. No need to reflash.
